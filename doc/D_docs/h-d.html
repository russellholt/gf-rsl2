D/C++ header dependencies

This graph below shows the order and extent of C++ #include dependencies in the core D framework. Because of the dual class representation, this diagram is very different from the D object model. It allows a much greater degree of self-describing flexibility than normally is feasible with standard C++ compilers that require complete class definitions. The #includes are almost the minimum possible. Any changes to this structure is likely to either be redundant, such as adding #include "DString.h" to just about anything because it is already included by D.h; or disasterous, resulting in recursively incomplete class definitions, such as changing the order of #include or adding, for example, #include "DClass.h" to the top of D.h. This does not represent a fragility of design, but rather, a highly self-referential design combined with static typing and very early binding in C++.
